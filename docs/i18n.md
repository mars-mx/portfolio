# MX Digital — i18n / Localization

> Tool-Recherche für native Lokalisierung (Deutsch + Englisch) in Frontend und Backend.
> Stand: **Juli 2026** (recherchiert gegen Next.js 16.2.9 / React 19.2.4 und FastAPI ≥0.137 / Pydantic v2 / Python 3.12).

---

## Entscheidung (TL;DR)

| Bereich  | Entscheidung | Zweite Wahl |
|----------|--------------|-------------|
| Frontend | **next-intl** (v4.13.x) | Paraglide JS (inlang) |
| Backend  | **Sprachneutrale Error-Codes**, Lokalisierung im Frontend; nur E-Mails serverseitig | starlette_babel (falls serverseitiger Text wächst) |
| API-Vertrag | **openapi-typescript** — Error-Codes als Enum in der OpenAPI-Spec, TS-Typen generiert | @hey-api/openapi-ts, orval |

---

## 1. Frontend: next-intl

### Ausgangslage

Next.js 16 hat für den App Router **kein eingebautes i18n-Routing** — das gab es nur
für den alten Pages Router. Die [offizielle App-Router-Doku](https://nextjs.org/docs/app/guides/internationalization)
beschreibt lediglich ein manuelles DIY-Pattern (`proxy.ts` für Locale-Erkennung,
`app/[lang]/`-Segment, Dictionaries per `getDictionary(lang)`) und verweist für
alles Weitere — lokalisierte Pfade, hreflang, typisierte Message-Keys — auf
Drittanbieter-Bibliotheken.

### Warum next-intl

De-facto-Standard für Next.js-App-Router-Projekte 2026, und die einzige
„batteries included“-Lösung, die alle Anforderungen dieses Projekts in einem
Paket abdeckt:

- **Lokalisierte Pfade:** `pathnames`-Config in `defineRouting` löst exakt
  `/ueber-mich` ↔ `/en/about`, mit typisierten `<Link>`-/`redirect`-APIs.
- **SEO:** `generateMetadata`- und hreflang-Unterstützung dokumentiert.
- **Statisches Rendering:** Standardmuster über `generateStaticParams` bleibt
  erhalten (kein Zwang zu dynamic rendering). Ab Next 16.2 entfällt dank
  Root Params sogar das `setRequestLocale`-Boilerplate.
- **PDF-Route-Handler:** `getTranslations()` ist eine awaitbare, hook-freie API,
  die explizit für Route Handler außerhalb des React-Baums dokumentiert ist —
  deckt `src/app/profil.pdf/route.ts` ab.
- **Wartung:** Sehr aktive Releases (Juli 2026), Next 16 + React 19 explizit
  bestätigt, größte Erprobungsbasis in der Community.

### Alternativen & Absagen

| Bibliothek | Status Juli 2026 | Bewertung |
|---|---|---|
| **Paraglide JS** (inlang, v2.20.x) | aktiv | Zweite Wahl. Kompilierte, typisierte Message-Funktionen (`m.hello()`), die ohne React-Kontext überall funktionieren; kleinere Bundles. Malus: Next.js-Integration deutlich DIY-lastiger (kein All-in-one-Paket, historischer Adapter-Churn). |
| **Lingui** (v6.4) | aktiv, aber blockiert | Abzuraten für diesen Stack: Babel-Macros kollidieren mit dem intern verwalteten React Compiler in Next 16 — laut [vercel/next.js#85313](https://github.com/vercel/next.js/discussions/85313) Stand Juni 2026 ungelöst. |
| **next-international** (v1.3.1) | letztes Release Okt. 2024 | Faktisch tot, nicht empfehlenswert. |
| **react-i18next / next-i18next** (v16) | aktiv | Reif, aber in Server Components spürbar mehr Boilerplate (kein Hook-Zugriff, manuelles Durchreichen); taucht in Next.js' eigener Empfehlungsliste nicht auf. |
| **Intlayer** (v8.x) | aktiv, aber klein | Content-Deklaration via typisierter `*.content.ts`-Dateien klingt nach unserem Muster, würde aber Umbau von `site.ts`/`profile.ts` in deren Dictionary-Format erzwingen; Route-Handler-Nutzung undokumentiert; Ein-Maintainer-Projekt (Bus-Factor). |

### Konsequenz für unser Content-Pattern

Keine der Bibliotheken verwaltet strukturierte Content-Objekte wie
`src/lib/profile.ts` / `src/lib/site.ts` — das bleibt in jedem Fall Handarbeit
als paralleles TS-Datenmodell, z. B.:

```ts
type LocalizedText = { de: string; en: string };
// oder: profile.de.ts / profile.en.ts mit identischem Type
```

next-intl übernimmt nur drei Dinge: (1) UI-Chrome-Strings (Nav, Buttons,
Formular-Labels, Footer), (2) Routing + lokalisierte Pfade + Metadata/hreflang,
(3) Locale-Auflösung im PDF-Route-Handler, um den richtigen Zweig aus den
Profildaten zu wählen.

## 2. Backend: sprachneutral — Error-Codes + Lokalisierung im Frontend

### Ausgangslage

FastAPI hat keine eingebaute i18n-Lösung und gibt keine offizielle Empfehlung
(vgl. [fastapi#7583](https://github.com/fastapi/fastapi/discussions/7583));
die Community löst es über Middleware + `contextvars`. Nutzersichtbarer Text
aus dem Backend: Validierungs-/Fehlermeldungen von `POST /api/contact`,
Statusmeldungen und perspektivisch die Bestätigungsmails (DE/EN).

### Entscheidung: Das Backend übersetzt nicht

API-Antworten enthalten **maschinenlesbare Error-Codes statt übersetzter
Texte**; die Lokalisierung übernimmt das Frontend über seine ohnehin
vorhandenen next-intl-Kataloge. Nur E-Mails werden serverseitig lokalisiert.
Gründe:

1. **Das Backend kennt die Anzeigesprache nicht zuverlässig.**
   `Accept-Language` ist die Browser-Einstellung, nicht die auf der Website
   gewählte Locale — nur das Frontend weiß sicher, welche Sprache der Nutzer
   gerade sieht.
2. **Kein zweiter Übersetzungs-Stack.** Mit next-intl existieren
   Message-Kataloge, Tooling und Typsicherheit bereits; Backend-Codes werden
   dort auf Keys gemappt. Neue Routen fügen nur neue Codes hinzu — kein
   paralleler `.po`-Workflow, der mit dem Frontend synchron zu halten wäre.
3. **Lokalisierte API-Responses skalieren schlecht:** Copy-Änderungen
   erfordern Backend-Deploys, Antworten sind ohne `Vary: Accept-Language`
   nicht cachebar, und jeder weitere Client (Mobile App, Dritt-Consumer)
   rendert Texte ohnehin selbst. Große APIs (Stripe, GitHub, …) liefern aus
   genau diesen Gründen Codes plus englische Entwickler-Message.

### Konkrete Umsetzung

1. Exception-Handler für `RequestValidationError`, der Pydantic-Fehler per
   `error['type']` auf stabile Codes mappt (z. B.
   `{"field": "email", "code": "email_invalid"}`) — Mapping-Pattern wie
   [offiziell in den Pydantic-Docs gezeigt](https://docs.pydantic.dev/latest/errors/errors/),
   nur mit Codes statt Texten als Ziel.
2. Das Frontend übersetzt die Codes über next-intl-Keys (z. B.
   `errors.email_invalid`); Erfolgsmeldungen rendert es ohnehin selbst.
3. E-Mails: Das Kontakt-Request-Schema bekommt ein `locale`-Feld (`de`/`en`),
   der Versand wählt eine von zwei Jinja2-Template-Varianten
   (`confirmation_de` / `confirmation_en`).

### Geprüfte Bibliotheken & Absagen

Serverseitige Übersetzungs-Bibliotheken wurden geprüft, aber mit der
Code-basierten Architektur für API-Responses grundsätzlich nicht benötigt.
Bewertung für den verbleibenden serverseitigen Text (E-Mails):

| Bibliothek | Status Juli 2026 | Bewertung |
|---|---|---|
| **starlette_babel** (v1.1.0) | Production/Stable, Release März 2026, aktiv | Zweite Wahl, falls der serverseitig gerenderte Text wächst (mehr als zwei Sprachen, viele E-Mail-Templates, Datums-/Zahlenformatierung): einzige aktiv gepflegte FastAPI/Starlette-spezifische Option, mit Locale-Middleware und Jinja2-Integration. |
| **fastapi-babel** (v1.0.0) | Alpha-Classifier, Ein-Personen-Projekt | Nicht empfohlen. |
| **pydantic-i18n** (v0.4.5) | letztes Release Sept. 2024, kaum gepflegt | Macht im Kern nichts anderes als das offizielle Pydantic-Dict-Pattern — Zusatznutzen rechtfertigt die Dependency nicht. |
| **python-i18n** (v0.3.9) | PyPI-Release von 2020 | Release-Pipeline de facto tot. |
| **fluent.runtime** (Mozilla) | Projekt aktiv | Mächtige Syntax (Plural, Genus), aber Lernkurve/Tooling lohnen erst bei komplexer Pluralisierung über viele Sprachen. |
| **gettext + Babel** | stdlib bzw. sehr aktiv | Solide, aber `.po`/`.mo`-Workflow (`pybabel extract/init/compile`) ist für 2 Sprachen und eine Handvoll Strings unverhältnismäßiger Tooling-Overhead. |

**Wechselkriterien zu `starlette_babel`:** betreffen nur serverseitig
gerenderten Text (v. a. E-Mails) — API-Responses bleiben auch bei Wachstum
Codes. Relevant ab: > 2 Sprachen, viele Templates, echter
Pluralisierungs-/Formatierungsbedarf, oder Anbindung eines Übersetzungs-Tools
(Lokalise, Phrase, Weblate) an `.po`-Dateien.

## 3. API-Vertrag & Client-Generierung (OpenAPI)

Damit das Frontend die Error-Codes typsicher nutzen kann, werden sie nicht als
Prosa dokumentiert, sondern **maschinenlesbar über die OpenAPI-Spec**
transportiert, die FastAPI ohnehin unter `/openapi.json` generiert.

**Ist-Zustand:** `src/lib/api.ts` ist ein handgeschriebener Fetch-Wrapper mit
manuell duplizierten Typen; es gibt noch keinen generierten Client.

### Die Kette

1. **Backend: Codes als Enum im Response-Modell.**

   ```python
   class ErrorCode(StrEnum):
       EMAIL_INVALID = "email_invalid"
       NAME_MISSING = "name_missing"
       MESSAGE_TOO_LONG = "message_too_long"

   class FieldError(BaseModel):
       field: str
       code: ErrorCode

   class ValidationErrorResponse(BaseModel):
       errors: list[FieldError]

   @router.post("/contact", responses={422: {"model": ValidationErrorResponse}})
   ```

   Das `responses={422: ...}` ist der entscheidende Handgriff: Ohne Override
   trägt FastAPI sein generisches `HTTPValidationError`-Schema ein. Mit dem
   Override landet das Enum als `enum: ["email_invalid", …]` in der Spec.
   Der `RequestValidationError`-Exception-Handler nutzt **dasselbe** Enum für
   sein Mapping — Spec und Laufzeitverhalten können nicht auseinanderlaufen.
   `Field(description=…)` ergänzt menschenlesbare Doku (sichtbar in `/docs`).

2. **Frontend: Typen aus der Spec generieren** mit `openapi-typescript`
   (reine TS-Typen, kein Runtime-Code — der schlanke handgeschriebene Wrapper
   bleibt):

   ```jsonc
   // package.json
   "gen:api": "openapi-typescript http://localhost:8210/openapi.json -o src/lib/api-types.d.ts"
   ```

3. **Erzwungene Vollständigkeit:** Aus dem Enum wird der Union-Type
   `"email_invalid" | "name_missing" | …`. Die Übersetzungszuordnung ist ein

   ```ts
   const errorMessageKeys: Record<ErrorCode, MessageKey> = { … }
   ```

   Damit prüft der Compiler die Kette Ende-zu-Ende: neuer Code im Backend →
   `pnpm gen:api` → TypeScript-Fehler, bis das Frontend eine Übersetzung
   dafür hat.

### Prozess

- Die generierte Typdatei (`src/lib/api-types.d.ts`) wird **eingecheckt**.
- Die Generierung braucht die Spec: entweder vom laufenden Backend
  (`make dev`) oder per Mini-Script, das `app.openapi()` ohne Server nach
  JSON dumpt (praktisch für CI).
- Alternativen (@hey-api/openapi-ts, orval) generieren komplette Clients bzw.
  React-Query-Hooks — für eine Route Overkill, bei Wachstum der API aber der
  natürliche Upgrade-Pfad.
